# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

Bare-metal firmware for the STM32 "Blue Pill" board — **STM32F103C8T6**, Cortex-M3, no FPU, 64K FLASH @ `0x8000000`, 20K RAM @ `0x20000000`. Generated by STM32CubeIDE's VS Code extension. There is no HAL/CMSIS driver layer — only the startup code, libc syscall/sysmem stubs, and `Src/main.c` (currently an empty `for(;;)` loop; the "blink" is not yet implemented).

## Build

Builds use CMake + Ninja with the `arm-none-eabi-` toolchain shipped in the STM32Cube bundle (`cube-cmake` is a wrapper that injects bundle paths — see `.vscode/settings.json`). Two presets: `Debug` (`-O0 -g3`) and `Release` (`-Os`).

```sh
cmake --preset Debug          # configure -> build/Debug/
cmake --build --preset Debug
```

Output lands in `build/<preset>/`: `blink_bluepill.elf`, plus `.hex` and `.bin` generated by a POST_BUILD objcopy step. A memory-usage summary and `arm-none-eabi-size` report print at the end of each build. The whole `build/` directory is gitignored.

Driving builds from outside VS Code requires `arm-none-eabi-*` on `PATH` and Ninja; the `cube-cmake`/`starm-clangd` wrappers assume `CUBE_BUNDLE_PATH` is set by the extension environment.

## Architecture & conventions

- **`CMakeLists.txt` is the user-editable build file** — add new source files to `sources_SRCS` and includes to `include_DIRS` here. Per the header comment it is generated once and never regenerated, so manual edits are safe.
- **`cmake/vscode_generated.cmake` is auto-generated — do not edit.** The IDE rewrites it. It appends the startup `.S`, `syscall.c`, `sysmem.c`, the linker script, and the `RTE_DEVICE_STARTUP_STM32F1XX` define. To change build inputs, use `CMakeLists.txt` instead.
- **CPU/FPU flags are derived, not hardcoded.** `CMakePresets.json` sets `CMSIS_D*` cache variables (core, fpu, endian, etc.); `cmake/gnu-tools-for-stm32.cmake` (the toolchain file) maps those to the actual `-mcpu=cortex-m3 -mthumb` flags and selects the compilers. To retarget a different MCU/core, change the `CMSIS_D*` values in the presets, not the compiler flags directly.
- **`stm32f103x8_flash.ld`** defines the memory map and is wired in as a `LINK_DEPENDS`, so editing it forces a re-link.
- Warnings are strict (`-Wall -Wextra -Wpedantic`); C11 and C++20 are both enabled. Float `printf` support is linked in via `-u _printf_float` (drop it if unused to save flash). Libc is `nosys` (no OS); syscalls are stubbed in `Src/syscall.c` / `Src/sysmem.c`.

## Notes

- No test harness, linter, or flashing target is configured in-repo — flashing/debugging is done through the STM32Cube VS Code extension (ST-Link / OpenOCD). See [docs/gravar-e-depurar.md](docs/gravar-e-depurar.md) for the full flash/debug workflow (CLI, OpenOCD, `launch.json`) and troubleshooting.
- This project nests under a same-named parent directory (`blink_bluepill/blink_bluepill/blink_bluepill`); the innermost dir is the actual project root and CMake source dir.
